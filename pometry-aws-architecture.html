1. Executive summary

This document describes the technical deployment architecture for Pometry (Raphtory) on AWS EC2, designed for a phased rollout: Phase 1 at 500 million nodes scaling to Phase 2 at 1.5 billion nodes. The architecture uses Pometry's on-disk (subscription) storage mode with high availability through active/passive failover across two Availability Zones, and an event-driven ingestion pipeline consuming change data from Databricks.

The graph models a financial services domain with three node types — Organizations (60%), Bankers (10%), and Clients (30%) — with Organization nodes forming deep hierarchical chains of up to 15 levels (HoldCo → SubGroup → Regional → Branch → … → leaf SPV). The on-disk strategy reduces infrastructure cost by 80–90% compared to pure in-memory deployment.

Key decisions: Pometry on-disk mode with Parquet on NVMe. Phase 1 on i4i.16xlarge (~$4K/mo per node), graduating to i4i.metal (~$8K/mo per node) at Phase 2. Ancestor chain materialisation for 15-level hierarchy queries. Stateless event ingestion from Databricks via MSK.

15-level Organisation hierarchy

Organisation nodes form deep parent-child hierarchies representing corporate ownership structures:

L0: Global HoldCo → L1: Regional Group → L2: Country Division → L3: Business Unit → L4–L13: Intermediate entities → L14: Leaf SPV / Branch

Hierarchy query implications

  • Upward traversal (UBO lookup): 15 hops from leaf to ultimate parent. Sub-millisecond when hot, <200ms cold on NVMe. Touches only 15 nodes.
  • Downward traversal (all subsidiaries): Fan-out from HoldCo can be exponential. With avg 4 children/level, worst case touches millions of nodes.
  • Recommended: materialised ancestor chains. Store full ownership path as property on each Org node. "Find all entities under EMEA" becomes a prefix filter instead of 15-hop recursive traversal.
  • Temporal hierarchy queries: Pometry natively supports "who owned this entity in March 2024?" Every PARENT_OF change is preserved in temporal history.

7. Layer 1 — Databricks event source

Databricks serves as the primary source of graph mutation events via Delta Live Tables with CDC.

Event schema

Event typeDescriptionKey payload fields
ADD_NODECreate/upsert a graph nodenode_id, node_type, properties, timestamp
ADD_EDGECreate a relationshipsrc_id, dst_id, edge_type, properties, timestamp
UPDATE_PROPERTIESModify node/edge propertiesentity_id, entity_type, properties, timestamp
DELETE_EDGERemove a relationshipsrc_id, dst_id, edge_type, timestamp
REPARENTChange PARENT_OF relationshipchild_id, old_parent_id, new_parent_id, timestamp
  • Delivery: Structured Streaming with foreachBatch → MSK via Kafka connector. Exactly-once via S3 checkpoints.
  • Schema management: Avro schemas in AWS Glue Schema Registry.
  • Connectivity: VPC peering or PrivateLink. No public internet.

9. Layer 3 — Graph event service

Stateless service consuming from MSK, validating, batching mutations, and writing to Pometry via GraphQL.

Service design

ComponentPhase 1Phase 2
RuntimePython 3.11+ (Raphtory SDK) or Node.js/Go with GraphQL client
HostingECS Fargate, 2 vCPU / 4 GB per task
Mutation consumers3 tasks3–6 tasks (auto-scaled)
Hierarchy processor1–2 dedicated tasks for REPARENT events
Batch size500–1,000 mutations per GraphQL call, flush every 2s

Hierarchy materialisation

When a REPARENT event arrives, a dedicated consumer queries Pometry for the affected subtree, recomputes the ancestors property for each descendant, and batches the updates back. Typically 1K–50K nodes per restructure. Processed asynchronously to not block the main pipeline.

Error handling & scaling

  • Retries: Exponential backoff (1s, 2s, 4s), max 3 per batch.
  • Circuit breaker: 5 consecutive failures → pause, alert PagerDuty. Auto-reset 60s.
  • Auto-scaling: ECS on Kafka lag. Out at >10K lag for 2 min. In at <1K for 10 min.

High availability

NVMe is ephemeral. Data lost on instance stop/terminate. Multi-layered backup mandatory.
Primary (AZ-1)

All read + write traffic via NLB. Incremental snapshots to EBS & S3 every 10 min. WAL streamed to S3 every 30–60s.

Standby (AZ-2)

Continuous S3 snapshot pull + WAL replay. Within 1–2 min of primary. Optional read-only queries. Promoted on failure.

Failover (RTO: 3–5 min)

Detection: 30s. Lambda → Step Functions. Standby stops replay, switches read-write. NLB target swap: 30s. New standby launched by ASG.

Backup (RPO: 30s–2 min)

WAL → S3 every 30–60s. Parquet snapshots → S3 every 10 min. EBS local backup. S3 cross-region for DR. Cold start: S3 ~13 min, EBS ~4 min.

13. Security considerations

  • Network isolation: Private subnets, no public IP. Access via internal NLB + VPC endpoints.
  • Encryption at rest: EBS with KMS. S3 with SSE-KMS. NVMe hardware-encrypted (Nitro).
  • Encryption in transit: TLS at ALB. Internal TLS where supported.
  • IAM: Least-privilege per component. EC2: S3 + CloudWatch only.
  • Connectivity: Databricks via VPC peering / PrivateLink.
  • Secrets: AWS Secrets Manager, rotated on schedule.
